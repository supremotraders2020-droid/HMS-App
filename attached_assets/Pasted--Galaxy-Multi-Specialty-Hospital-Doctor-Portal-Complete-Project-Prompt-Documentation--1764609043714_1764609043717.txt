# Galaxy Multi Specialty Hospital - Doctor Portal

## Complete Project Prompt & Documentation

---

## Project Overview

Build a comprehensive **Hospital Management System (HMS) Doctor Portal** for "Galaxy Multi Specialty Hospital" located at Sane Chowk, Nair Colony, More Vasti, Chikhali, Pimpri-Chinchwad, Maharashtra 411062.

This web application serves as a dedicated interface for medical professionals (doctors) to manage their daily clinical operations. The system follows a **medical-first design approach** with professional aesthetics specifically tailored for healthcare environments.

---

## Target Users

- **Primary Users**: Doctors and Medical Professionals
- **Use Case**: Daily hospital operations management
- **Environment**: Hospital/Clinic desktop and mobile access

---

## Core Features

### 1. Authentication & Login
- Secure login page with hospital branding
- Username and password authentication
- "Remember me" functionality
- Session management with 24-hour expiry
- Demo mode for design presentations (any credentials work)

### 2. Dashboard
- **KPI Cards**: Display key statistics
  - Today's Appointments count
  - Pending Prescriptions count
  - Total Patients count
  - Weekly Trend indicators
- **Today's Appointments List**: Quick view of scheduled appointments
- **Recent Notifications**: System alerts and updates
- **Quick Actions**: Fast access to common tasks

### 3. Patient Management
- **Patient List View**: Searchable, filterable patient records
- **Patient Cards**: Display patient info including:
  - Name, Age, Gender
  - Blood Group (with color-coded badge)
  - Phone, Email, Address
  - Emergency Contact
- **CRUD Operations**: Add, Edit, View, Delete patients
- **Search**: Filter by name, phone, or email

### 4. Appointment Management
- **Appointment List**: View all appointments
- **Appointment Cards**: Show patient name, date, time, status
- **Status Types**: Pending, Confirmed, Completed, Cancelled
- **Appointment Details**: Complaint, Notes, OPD assignment
- **CRUD Operations**: Create, update, cancel appointments

### 5. Schedule Management
- **Weekly Schedule**: Doctor's availability by day
- **Time Slots**: Start time, End time for each day
- **OPD Assignment**: Assign to specific OPD departments
- **Availability Toggle**: Mark days as available/unavailable

### 6. Prescription Management
- **Prescription List**: View all prescriptions
- **Prescription Details**:
  - Patient information
  - Diagnosis
  - Medicines (array of medications)
  - Instructions
  - Date issued
- **Create Prescriptions**: Link to appointments

### 7. Notifications
- **Notification Types**: Appointment, Patient, System
- **Read/Unread Status**: Visual indicators
- **Notification Actions**: Mark as read, Dismiss
- **Time Display**: Relative time (e.g., "2h ago")

### 8. Doctor Profile
- **Personal Information**: Name, Specialty, Contact
- **Professional Details**: Qualifications, Experience
- **Working Hours**: Schedule overview
- **Profile Management**: Update personal information

## Database Schema

### Users (Doctors)
```typescript
{
  id: varchar (UUID, primary key),
  username: text (unique, required),
  password: text (hashed, required),
  name: text (required),
  specialty: text (required),
  email: text (required),
  phone: text (required)
}
```

### Patients
```typescript
{
  id: varchar (UUID, primary key),
  name: text (required),
  age: integer (required),
  gender: text (required),
  phone: text (required),
  email: text (optional),
  address: text (optional),
  bloodGroup: text (optional),
  emergencyContact: text (optional)
}
```

### Appointments
```typescript
{
  id: varchar (UUID, primary key),
  patientId: varchar (foreign key, required),
  doctorId: varchar (foreign key, required),
  date: text (YYYY-MM-DD, required),
  time: text (HH:MM, required),
  status: text (pending|confirmed|completed|cancelled),
  complaint: text (optional),
  notes: text (optional),
  opd: text (optional)
}
```

### Schedules
```typescript
{
  id: varchar (UUID, primary key),
  doctorId: varchar (foreign key, required),
  day: text (monday-sunday, required),
  startTime: text (HH:MM, required),
  endTime: text (HH:MM, required),
  opd: text (optional),
  isAvailable: boolean (default: true)
}
```

### Prescriptions
```typescript
{
  id: varchar (UUID, primary key),
  patientId: varchar (foreign key, required),
  doctorId: varchar (foreign key, required),
  appointmentId: varchar (optional),
  date: text (YYYY-MM-DD, required),
  medicines: text[] (array of medicines),
  instructions: text (optional),
  diagnosis: text (optional)
}
```

### Notifications
```typescript
{
  id: varchar (UUID, primary key),
  doctorId: varchar (foreign key, required),
  title: text (required),
  message: text (required),
  type: text (appointment|patient|system, required),
  isRead: boolean (default: false),
  createdAt: text (ISO timestamp, required)
}
```

## Authentication Flow

1. User visits application
2. System checks for existing session (`/api/auth/me`)
3. If no session → Show Login page
4. User enters credentials
5. POST to `/api/auth/login`
6. On success → Create session, redirect to Dashboard
7. On failure → Show error message
8. Logout → Destroy session, redirect to Login

### Demo Mode
- For design/demo purposes, any username/password works
- Creates temporary session with entered username
- User appears as "Dr. [username]"

## Data Flow

### React Query Pattern
```typescript
// Fetching data
const { data, isLoading, error } = useQuery({
  queryKey: ['/api/patients'],
  queryFn: async () => {
    const response = await fetch('/api/patients');
    if (!response.ok) throw new Error('Failed to fetch');
    return response.json();
  }
});

// Mutations
const mutation = useMutation({
  mutationFn: async (newPatient) => {
    return apiRequest('/api/patients', { method: 'POST', body: newPatient });
  },
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['/api/patients'] });
  }
});
```
